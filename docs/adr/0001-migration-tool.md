# ADR 0001: Relational Migration Tool

- Status: Accepted
- Date: 2026-04-18
- Owners: architecture / platform
- Deciders: DevQ&A team
- Related tasks: [S0-REPO-03](../planning/00-foundation.md#s0-repo-03--decide-and-document-migration-tool)

## Context

This repo will ship multiple NestJS services, each with its own PostgreSQL database and its own migration history. Sprint 1 alone introduces `user-service`, `tag-service`, `question-service`, and `answer-service`, and later sprints add more relational services.

The migration tool must work cleanly with:

- Node.js 20, TypeScript 5.x, and pnpm workspaces
- NestJS services in `packages/services/<service>`
- one schema history per service, committed to Git
- local development, CI, and Docker Compose deployment flows
- explicit SQL review for production-facing schema changes

The candidates considered in the planning docs are Prisma, TypeORM, and Knex.

## Decision

We will standardize on **Prisma Migrate** for every relational service in this repo.

Each service that owns a PostgreSQL database will use:

- `prisma/schema.prisma` as the canonical service-local data model
- `prisma/migrations/*` committed to source control
- `prisma migrate dev` in local development
- `prisma migrate deploy` in CI / container startup / deployment workflows

When Prisma schema DSL cannot express a database-specific feature directly, the team will keep Prisma as the system of record and use a customized SQL migration generated by Prisma Migrate rather than introducing a second migration tool for that service.

## Why Prisma

### Best fit for this repo's service model

- Each service owns its own database, so a per-service `prisma/` folder and migration history maps naturally to the repo layout.
- Prisma's declarative schema keeps the migration source of truth close to the service model instead of spreading DDL across handwritten files from day one.
- Generated SQL migrations remain reviewable and can be customized when needed.

### Better developer ergonomics for a class-scale NestJS codebase

- Prisma gives the team a fast path from schema definition to typed client usage in TypeScript.
- The schema file is easier to review in early iterations than entity-decorator metadata spread across many files.
- The migration workflow is straightforward enough for every service team to apply consistently.

### Better default discipline than TypeORM sync-style workflows

- TypeORM migrations are viable, but the ecosystem still makes it easy to drift toward entity-first changes plus migration generation as an afterthought.
- This project benefits from a stronger "schema first, migration history committed, deploy with explicit migration command" workflow.

## Options Considered

### Prisma

- Pros:
  - Declarative schema with generated SQL migrations
  - Strong TypeScript ergonomics for application code
  - Clean fit for one-database-per-service ownership
  - Explicit development (`migrate dev`) and deployment (`migrate deploy`) workflows
- Cons:
  - Some PostgreSQL features are not modeled directly in Prisma schema
  - Database-specific edge cases may require editing generated SQL

### TypeORM

- Pros:
  - Mature NestJS integration and familiar entity/decorator model
  - Migrations support raw SQL and complex PostgreSQL features directly
- Cons:
  - More temptation to couple persistence entities, runtime ORM behavior, and schema evolution too tightly
  - Generated migrations are typically less predictable and less pleasant to review than Prisma's schema-to-SQL flow for this project style

### Knex

- Pros:
  - Lowest abstraction and full SQL control
  - Good migration primitives and locking model
- Cons:
  - Too low-level for the default path in a repo with many services and many student/developer contributors
  - Pushes more schema modeling, typing, and consistency work onto each service team
  - We would still need to make additional choices for typed data access in each service

## Consequences

### Positive

- Every relational service follows the same folder layout and migration commands.
- Sprint 1 services can use one repeatable bootstrap pattern instead of evaluating tooling per service.
- CI and deployment can standardize on a Prisma migration step before service startup or release.

### Negative

- The repo accepts Prisma-specific conventions and generated artifacts as part of the baseline.
- Some advanced PostgreSQL constructs may require customized SQL within a Prisma-generated migration.
- Switching later to TypeORM or Knex would be expensive because migration histories and service templates would already be Prisma-shaped.

## Rules Going Forward

1. Do not introduce TypeORM migrations or Knex migrations in a new relational service.
2. Every relational service keeps its own `prisma/` directory and migration history inside that service package.
3. Use `prisma migrate dev` for local schema evolution and `prisma migrate deploy` outside local development.
4. If a service needs a PostgreSQL feature not fully represented in Prisma schema, implement it by customizing the generated SQL migration and document the exception in that service.
5. Keep migration history in Git; never rely on `db push` as the production migration mechanism.

## Notes for Sprint 1

- The `answer-service` requirement for a partial unique constraint (`where accepted_at is not null`) may require customized SQL in the generated migration.
- That is acceptable within this decision and does not justify introducing TypeORM or Knex for only one service.
